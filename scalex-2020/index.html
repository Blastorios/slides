Traefik Project

Traefik Core Concepts

Traefik is an Edge Router

  • Traefik sits between the internet and your platform

  • It is on of the first compoment handling the requests coming to your cluster

  • And is in charge of routing them

Dynamically Discovers Services

  • It listens to an external source of configuration, called a provider

  • It updates its routing confguration without any restarts, in ANY situation

  • It doesn’t loose any requests doing this (THANK YOU GOLANG)

Architecture at a Glance

  • DO NOT DIVE INTO DETAILS RIGHT NOW !

  • Enumerate core concepts and that’s it.

Entrypoints

  • Entrypoints defines which ports are being listened on by Traefik

  • Carries all the network listening configuration (like allowed timeouts)

Routers

  • Routers matches an incoming request with a service according to rules.

  • Both with HTTP and TCP.

  • HTTP routes according to request metadata

  • TCP routes according to the entrypoint OR the SNI in case of TLS traffic.

  • You can multiplex TLS traffic onver the same port with Traefik !

Middlewares

  • Pieces of software in charge of tweaking the incoming / outgoing requests.

  • Rewrite request path

  • Perform redirections

  • Rate limit ?

  • Authentication

Services

  • Responsible for configuring how to reach the actual services.

  • Server Addresses !

  • How do we want to loadbalance ? WRR ? Do we want to mirror traffic ?

Architecture (again) at a Glance

  • To sum up

  • Request arrivethrough an entrypoint

  • ..Is matched by a router

  • ..Is tweaked by some middlewares attached to this router

  • ..Is forwared to a service that does the loadbalancing and sent to an actual server.

Static vs Dynamic

  • Static is the startup configuration of Traefik, and can’t be updated without a restart.

  • Dynamic is the actual routing configuration and is derived from the provider.

Traefik and Let’s Encrypt

HTTPS & Let’s Encrypt

The Automatic Certificate Management Environment

Let’s encrypt HTTP Challenge

Pros:

  • It’s easy to automate without extra knowledge about a domain’s configuration.

  • It works with off-the-shelf web servers.

Cons: - It doesn’t work if your ISP blocks port 80 (this is rare, but some residential ISPs do this). - Let’s Encrypt doesn’t let you use this challenge to issue wildcard certificates. - If you have multiple web servers, you have to make sure the file is available on all of them

Let’s encrypt TLS-ALPN Challenge

Relies on TLS-ALPN : Application Layer Protocol Negociation

Pros: - It works if port 80 is unavailable to you. - It can be performed purely at the TLS layer.

Cons: - It’s not supported by Apache, Nginx, or Certbot, and probably won’t be soon. - Like HTTP-01, if you have multiple servers they need to all answer with the same content.

Let’s encrypt DNS Challenge

Pros: - You can use this challenge to issue certificates containing wildcard domain names. - It works well even if you have multiple web servers.

Cons: - Keeping API credentials on your web server is risky. - Your DNS provider might not offer an API. - Your DNS API may not provide information on propagation times.

Ingress Example with ⎈

kind: Ingress
metadata:
  name: corporate-webapp
  annotations:
    kubernetes.io/ingress.class: 'traefik'
spec:
  rules:
  - host: localhost
    http:
      paths:
      - backend:
          serviceName: corporate-webapp
          servicePort: 80

⎈ Ingress Route (HTTP)

webapp.yaml

apiVersion: traefik.containo.us/v1alpha1
kind: IngressRoute
metadata:
  name: myapp
spec:
  entryPoints:
    - web
  routes:
  - match: Host(`localhost`) && PathPrefix(`/whoami`)
    services:
    - name: webapp
      port: 80

    - mongotcp
